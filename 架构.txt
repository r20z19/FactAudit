Cairn 项目深度架构分析
一、项目概述
Cairn（衍迹）是一个基于黑板架构（Blackboard Architecture）的通用的目标导向问题求解引擎，以渗透测试为主要验证场景。版本 0.2.1，AGPLv3 许可，纯 Python 实现。
核心理念：将目标导向的探索过程建模为有向无环超图——从已知事实出发，不断探索产出新事实，直到目标达成。"Cairn" 一词源于登山者用石头垒起的路标。
二、四层架构
┌─────────────────────────────────────────────────────────────────┐
│ 1. Cairn Server (FastAPI + SQLite)                              │
│    协议真相源：维护 Fact-Intent-Hint 图，仅做一致性校验          │
├─────────────────────────────────────────────────────────────────┤
│ 2. Cayn Dispatcher (Python 调度引擎)                            │
│    唯一的客户端写入者：读图 → 决策派发什么任务 → 找 Worker 执行  │
│    → 把结果写回 Server                                           │
├─────────────────────────────────────────────────────────────────┤
│ 3. Project Container (Docker, 每个项目一个)                      │
│    执行环境：Kali Linux + 100+ 安全工具 + 3 个 LLM 编码 Agent    │
├─────────────────────────────────────────────────────────────────┤
│ 4. Worker / Agent CLI (claude / codex / pi)                     │
│    LLM 大脑：接收 Markdown Prompt → 用工具执行命令 → 返回 JSON   │
└─────────────────────────────────────────────────────────────────┘
三、黑板交互机制 —— Fact-Intent 图（核心）
3.1 数据模型
概念	数据库表
Fact	facts
Intent	intents + intent_sources
Hint	hints
3.2 Fact 的设计
origin ───(i001,bootstrap)───→ f001 ────┐
                                         ├──(i003,exploit SQL)───→ f003
f002 ────────────────────────────────────┘
- 两个特殊 Fact 在项目创建时写入：origin（起点/问题描述）和 goal（终点/期望目标）
- 普通 Fact 由系统生成 ID（f001, f002...）
- 状态变化通过追加新 Fact 表达，而非修改历史。例如 f003:"获得 root shell" → 后来 f025:"shell 已断开"
- Fact 的 description 应有结论性摘要，原始大数据（如 nmap 扫描结果）保留在容器文件系统中，描述里只写引用路径
3.3 Intent 的设计 —— 超边
class Intent:
    id: str                    # i001, i002...
    from_: list[str]           # 源 Fact ID 列表（支持超边：多个源节点→一个目标）
    to: str | None             # 结论 Fact ID；None = 尚未结论
    description: str           # 探索方向描述
    creator: str               # 谁提出的方向（不可变）
    worker: str | None         # 当前谁在执行（claim 机制）
    last_heartbeat_at: str     # 最后心跳时间
    concluded_at: str | None   # 完成时间
Intent 的三种状态和 claim 租约机制：
状态	to
未认领	None
执行中	None
已结论	非空
并发控制：
- Intent 级：worker 字段 + 心跳（默认 15 秒超时）。同一 Intent 最多一个 worker。
- 项目级 Reason：project.reason lease，确保同一时刻只有一个 worker 做全局推理。
- Bootstrap 单例：设计上每个项目最多一个 bootstrap intent。
3.4 Hint 的设计
Hint 是外部旁注，不参与图的因果推理，只对 Agent 起建议/指导作用：
class Hint:
    id: str         # h001, h002...
    content: str    # 提示内容
    creator: str    # 谁提供的
    created_at: str
- 可在项目创建时或运行中随时添加
- 对 stopped 和 completed 项目也可写入
- 起到类似军事指挥中"指挥官意图"（Mission Command）或蚁群信息素的作用
四、Docker 容器调用机制
4.1 整体架构
Dispatcher 通过挂载 Docker socket（/var/run/docker.sock）使用 docker Python SDK 管理容器：
# docker-compose.yaml
cairn-dispatcher:
  volumes:
    - /var/run/docker.sock:/var/run/docker.sock  # 核心：Docker 通信通道
    - ./dispatch.yaml:/etc/cairn/dispatch.yaml
4.2 容器生命周期（ContainerManager, containers.py）
创建模式：每个项目一个容器，命名为 cairn-dispatch-{project_id}
# 核心方法 ensure_running()
container = docker.from_env().containers.run(
    image="ghcr.io/oritera/cairn-worker-container:latest",
    command=["sleep", "infinity"],     # 关键：保持容器存活
    detach=True,
    name="cairn-dispatch-{project_id}",
    network_mode="host",               # 使用宿主机网络
)
- 复用：同一项目的多次任务复用同一个容器（检查是否已存在且 running）
- sleep infinity：容器主进程始终是 sleep infinity，实际工作通过 docker exec 执行
- 网络模式：默认 host 模式，渗透工具需要与实际网络交互
- 无卷挂载：文件通过 tar 流传入（put_archive），不在 host 和 container 间共享文件系统
- 完成后的清理：completed_action 可选 stop（保留容器供检查）或 remove（彻底删除）
4.3 文件写入：tar 流注入
容器内没有挂载的卷，所有文件通过 内存中构建 tar 归档 → container.put_archive() 写入：
def write_text_file(container, path, content):
    # 1. 在内存中构建 tar 归档（Python tarfile 模块）
    archive = _text_file_archive(path, content)
    # 2. 通过 Docker SDK 上传并解压到容器内
    container.put_archive(archive_path, archive_bytes)
典型用法：在任务执行前，将图快照 YAML写入容器 /tmp/cairn-prompts/<phase>-<uuid>/graph.yaml。
4.4 进程执行：ManagedProcess（process.py）
# 1. 构建命令（可选 timeout 包装）
command = ["timeout", "-k", "5s", "600s"] + actual_command

# 2. 通过 Docker API 创建 exec 实例（不分配 TTY，无 stdin）
exec_info = api.exec_create(container_id, command, 
                            stdout=True, stderr=True, 
                            stdin=False, tty=False)

# 3. 启动 exec 并以 demux 流式读取 stdout/stderr
stream = api.exec_start(exec_id, stream=True, demux=True)
# 每个 chunk = (stdout_bytes, stderr_bytes) 元组
关键设计：
- 无 stdin 交互：Agent 不通过 stdin 输入，全部通过命令行参数 -p "prompt" 传入
- demux=True：stdout 和 stderr 多路分离，分别收集
- timeout 包装：使用 Linux timeout 命令双重保险（-k 5s = 超时后再给 5 秒然后 SIGKILL）
- 取消机制：两层保证——心跳丢失时 kill 进程，项目状态变为非 active 时 cancel 进程
4.5 容器镜像（container/Dockerfile）
基于 kalilinux/kali-rolling:latest，预装：
- 安全工具：bloodyad, pwncat, netexec, nuclei, naabu, katana, dalfox, kerbrute, cloudfox, nikto, dirsearch 等
- 脚本/语言：npm + playwright-cli, python3 + pwntools, sshpass, ncat
- 知识库：PayloadsAllTheThings, HackTricks, HackTricks Cloud 等
- 漏洞库：CVE-PoC, exphub, vulhub 等
- 3 个编码 Agent（全局 npm 安装）：@anthropic-ai/claude-code, @openai/codex, @mariozechner/pi-coding-agent
五、AI 工具调用协议 —— "JSON in stdout" 模式
5.1 核心结论：不是 API 调用，不是 Regex 匹配文本，而是 CLI 命令 + JSON 解析
Cairn 不是通过 API 直接调用 LLM，而是：
1. 用命令行参数向编码 Agent CLI 传递 Markdown Prompt
2. Agent 在容器内执行命令、读文件、编辑代码
3. Agent 在 stdout 输出结构化 JSON
4. Dispatcher 用 output_parser.py 从 stdout 提取 JSON 对象
5. 用 contracts.py 校验 JSON 结构
5.2 三种 Agent Driver 的具体 CLI 命令
Driver	执行命令
ClaudeCode	claude --session-id <uuid> --dangerously-skip-permissions -p -- "<prompt>"
Codex	codex exec --dangerously-bypass-approvals-and-sandbox --model $MODEL -c ... -- "<prompt>"
Pi	/bin/sh -lc '...' -- /tmp/cairn-pi/<worker> <models.json> <pi-flags>
Mock	python3 -c "<script>" <behavior.json> <prompt.json>

class RegexSessionDriver(WorkerDriver): # Codex 使用
    session_pattern = re.compile(r"session id:\s*([0-9a-fA-F-]+)")
    def extract_session(session, stdout, stderr):
        # 从 stderr 中用正则提取 session id
Session 的作用：当阶段一执行失败/超时时，用同一 session 进入 Conclude 回退模式（bootstrap_conclude.md / explore_conclude.md），让 LLM 立刻汇总已确认结果，避免信息完全丢失。
5.4 JSON 提取器（output_parser.py / extract_json_object()）
AGENT 可以任意形式输出 JSON，提取器按以下顺序尝试：
1. 直接 json.loads(全文) → 成功则返回
2. 正则匹配 markdown fenced code block： `(?:json)?\s*\n?(.*?)``` `` → 尝试解析
3. 扫描每个 { 字符位置，用 JSONDecoder().raw_decode() 尝试找到有效 JSON 对象
5.5 Contract 校验（contracts.py）
每种任务对应一种 JSON 结构约束：
任务	有效输出
Reason	{"data":{"complete":{"from":["f003"],"description":"..."}}}
 	{"data":{"intents":[{"from":["f001"],"description":"exploit SQLi"}]}}
 	{"data":{}}
 	{"accepted":false}
Explore	{"data":{"description":"nmap found port 22,80,443 open"}}
Bootstrap	{"data":{"fact":{"description":"..."},"complete":{"description":"..."}}}
还支持外层包装格式 {"accepted":true, "data":{...}}，若 accepted 缺失则接受裸 payload。

七、三种任务类型详解
7.1 Bootstrap —— 初始一把梭
触发条件：项目初始态（仅有 origin + goal 两个 Fact）
Prompt：bootstrap.md，接收 {origin}, {goal}, {hints}
工作流：
Phase 1 (execute): Agent 直接尝试解决整个问题
  ├─ 成功 → 输出 {"fact": {...}, "complete": {...}}
  │         Dispatcher: conclude(fact) → complete(from=[fact_id])
  └─ 失败/超时 → Phase 2 (conclude fallback)
                  → 用同一 session 进入 bootstrap_conclude.md
                  → 仅输出 fact（禁止 complete）
                  → Dispatcher: 仅 conclude(fact)，不标记完成
7.2 Reason —— 读图判断，提出方向
触发条件（_reason_trigger() 检查 checkpoint 差异）：
- 新 Fact 产生
- 新 Hint 添加
- 所有 open intents 都已结论（一轮探索结束）
Prompt：reason.md，接收 {graph_yaml}, {fact_ids}, {open_intents}, {max_intents}
返回三种结果之一：
1. complete → 项目完成
2. intents: [...] → 提出新的探索方向（每条 intent 包含 from 源事实列表和 description）
3. noop → 当前没有新的有价值方向
并发：项目级 Reason lease 确保同时最多一个 Reason 任务。
7.3 Explore —— 认领 Intent，执行产出事实
触发条件：存在未认领的 Intent（worker=None, to=None）
Prompt：explore.md，接收 {graph_yaml}, {intent_id}, {intent_description}
工作流：
Phase 1 (execute): Agent 执行探索，产出 fact description
  ├─ 成功 → 输出 {"data": {"description": "..."}}
  │         Dispatcher: conclude(intent) → 自动创建新 Fact
  └─ 失败/超时 → Phase 2 (conclude fallback)
                  → explore_conclude.md → 立即总结已确认结果
                  → Dispatcher: 仍 conclude，保留部分成果
并发：每个项目可同时运行多个 Explore 任务（受 max_project_workers 限制）。
八、调度主循环（DispatcherLoop.run()）
while True:
    reap_futures()          # 收已完成的任务结果
    reap_cleanup_futures()  # 收容器清理结果
    summaries = client.list_projects()  # 拉取所有项目
    
    init_reason_checkpoints()            # 建立原因触发 checkpoint
    refresh_runtime_projects()           # 跟踪活跃项目
    cancel_inactive_tasks()              # 取消非 active 项目的任务
    queue_container_cleanups()           # 排队清理完成/停止项目的容器
    
    dispatch_available(summaries)        # 核心：决定派发什么任务
    
    sleep(interval)                      # 等待下一轮
派发决策树（_try_dispatch_project）：
活跃项目 → 
  ├── 初始态 + bootstrap_enabled → dispatch BOOTSTRAP
  ├── 初始态 + !bootstrap_enabled  → dispatch REASON
  ├── reason_trigger 返回非空     → dispatch REASON
  ├── 有未认领 intent              → dispatch EXPLORE（选最新的）
  └── 否则 → SKIP
Worker 选择（choose_worker）：
1. 过滤 task_type 兼容的
2. 过滤已达 max_running 上限的
3. 过滤不健康的（冷却期内）
4. 过滤被拒绝的（(project, task_type, worker) 三元组维度）
5. 按 priority + 当前运行数排序 → 选最优的

十、数据流全链路示例
1. 用户 POST /projects {origin:"渗透目标A", goal:"获取Domain Admin权限", hints:[...]}
   → Server 创建项目，创建 origin+goal 两个 Fact，创建 hints

2. Dispatcher 轮询发现新项目（初始态）
   → dispatch BOOTSTRAP → 选中 claudecode worker
   → ensure_running(project_id) → Docker 容器启动（sleep infinity）
   → write graph.yaml 到容器 /tmp/cairn-prompts/
   → exec: claude --session-id <uuid> -p "<bootstrap.md prompt>"
   → Agent 尝试直接攻击目标...

3. Agent 返回 JSON，bootstrap 完成
   → Dispatcher 调用 Server: conclude(intent, fact_description) → 新 Fact f001
   → 如果同时返回 complete → Server 标记项目 completed

4. 如果 bootstrap 未完成 → Reason 触发
   → dispatch REASON → 选中另一个 worker
   → exec: claude -p "<reason.md prompt with graph YAML>"
   → Agent 读图、读 hints → 返回 {"intents": [...多个探索方向...]}
   → Dispatcher 调用 Server: create_intent(...) × N

5. Explore 触发（未认领 Intent 存在）
   → dispatch EXPLORE × 多个（并发）
   → 每个 Agent: exec tool + explore → 返回 {"description": "发现..."}
   → Dispatcher: conclude(intent) → 新 Fact

6. 回到步骤 4（Reason 因新 Fact 再次触发）→ 循环
   → 直到某个 Reason 轮次判断目标达成 → complete → 项目 done
这就是 Cairn 的完整架构。核心创新点在于：把渗透测试建模为黑板图上的 OODA 搜索过程，LLM 是图中的"专家"，通过 JSON-on-stdout 协议与调度器解耦，Hint 提供人机协同的旁路通道。整个系统对 Agent 来说是一个"读图、思考、执行、写回"的循环。
