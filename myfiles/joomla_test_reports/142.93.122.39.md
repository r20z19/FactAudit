## Vulnerability 5: com_finder Reflected XSS - Detailed Analysis

| Check | Result |
|-------|--------|
| Finder accessible | ✅ |
| query-explained block | ✅ Present |
| span.term reflected | ✅ Search term reflected in page |
| WAF blocks &lt;script&gt; | ✅ No WAF block |

**Conclusion**: The com_finder component reflects the search term inside a query-explained block with <span class="term"> without htmlspecialchars encoding. However, the default findermin tokenizer strips HTML special characters (< > \" ' = ( ) { } &), making practical XSS exploitation infeasible.

## Key Findings

1. **Registration**: Closed - all auth-requiring vulns blocked
2. **API**: API endpoints exist but require authentication tokens
3. **com_finder XSS (Vuln 5)**: Code-level confirmed (unencoded span.term output). Exploitation limited by default tokenizer HTML stripping.
4. **All authenticated vulns**: Blocked by registration closure and/or missing credentials

## Conclusion

All 13 vulnerabilities are theoretically applicable to Joomla 5.4.6 but **none are practically exploitable** without valid credentials. The com_finder reflected XSS (Vuln 5) is the only one approachable without authentication but is neutered by the default tokenizer.
